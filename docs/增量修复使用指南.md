# 🚀 增量修复功能使用指南

## 问题背景

在批量处理 31 本哲学书时，发现部分书籍质量检查不达标，但当前系统会直接抛出异常阻止写入，导致：
1. 需要全部重新生成（耗时 + API 成本）
2. 已合格的部分也被丢弃
3. 无法定向修复问题字段

## 解决方案

实现"质量不达标先落盘再增量修复"机制：
- ✅ 质量不达标时**仍写入结果文件**
- ✅ 自动生成**修复清单**（JSON + Markdown）
- ✅ **增量修复**：只修复问题字段，不重新生成整本书
- ✅ **优先级驱动**：CRITICAL → HIGH → MEDIUM

## 使用流程

### 1. 批量处理书籍

```bash
python main.py --input "/path/to/books/" --batch
```

系统会自动：
- 解析 + 分析 + 生成图谱
- 质量检查
- **质量不达标时仍落盘**，并生成修复清单

### 2. 查看修复清单

修复清单保存在 Obsidian vault 的 `.repair_manifests/` 目录：

```bash
# 查看所有待修复书籍
ls .repair_manifests/*_repair_manifest.md

# 查看具体书籍的修复清单
cat .repair_manifests/善恶的彼岸_repair_manifest.md
```

**修复清单示例**：

```markdown
# 📋 修复清单：善恶的彼岸

**生成时间**：2026-06-19
**质量评分**：0/100
**问题总数**：10

## 🚨 需修复项（按优先级排序）

### 🔴 CRITICAL（必须修复）

#### 1. 章节合并

- **字段路径**：`chapters[chapter_number='11-22']`
- **问题描述**：章节编号 '11-22' 为合并编号，LLM偷懒行为
- **当前值**：`第11-22章：...`
- **建议操作**：拆分为独立章节，逐章分析
```

### 3. 批量增量修复

```bash
# 批量修复所有待修复书籍
python batch_repair.py --manifest-dir ".repair_manifests/"
```

系统会：
1. 扫描所有修复清单
2. 按优先级定向修复（CRITICAL → HIGH → MEDIUM）
3. 重新质量检查
4. 更新修复清单状态

### 4. 验证修复结果

修复完成后，系统会：
- 移除输出文件中的修复标记
- 更新修复清单状态为 `completed` 或 `partial`
- 重新生成质量报告

## 修复策略详解

### CRITICAL 级别（必须修复）

| 问题类型 | 修复策略 |
|---------|---------|
| **章节合并** | 拆分合并章节，逐章重新生成 |
| **章节占位符** | 重新生成章节内容（如"TBD"、"待补充"） |
| **全局占位符污染** | 全局扫描并替换占位符 |

### HIGH 级别（高优先级）

| 问题类型 | 修复策略 |
|---------|---------|
| **章节覆盖率不足** | 补充缺失章节 |
| **概念占位符** | 补充概念定义和深层含义 |

### MEDIUM 级别（中优先级）

| 问题类型 | 修复策略 |
|---------|---------|
| **浅薄概念** | 扩展概念定义（< 30 字符） |
| **金句不足** | 从原文提取更多金句 |
| **底层逻辑格式错误** | 重写为标准格式（前提假设 → 推理链条 → 核心结论） |

## 配置说明

在 `config.yaml` 中启用增量修复：

```yaml
improvements:
  quality_gate:
    enabled: true
    threshold: 80.0              # 质量分数阈值
    auto_retry: true             # 自动重试
    max_retries: 3               # 重试次数
    lazy_write: true             # 🔑 质量不达标时仍落盘
    generate_repair_manifest: true  # 🔑 自动生成修复清单
```

## 优势对比

| 方案 | 旧方案（Fail-Fast） | 新方案（Lazy-Write） |
|------|------------------|-------------------|
| **质量不达标时** | 抛异常，阻止写入 | 先落盘，生成修复清单 |
| **已合格内容** | 被丢弃 | 保留 |
| **修复方式** | 全部重新生成 | 定向修复问题字段 |
| **API 成本** | 高（全部重生成） | 低（只修复问题部分） |
| **时间成本** | 高 | 低 |

## 实际案例

### 善恶的彼岸（质量评分 0/100）

**问题清单**：
- 🔴 章节覆盖率不足：10/17 (59%)
- 🔴 占位符污染严重：45 处
- 🟠 概念含占位符：6 个
- 🟡 金句数量不足：3 条（达标）
- 🟡 底层逻辑格式：80%（达标）

**修复策略**：
1. **CRITICAL**：全局清理 45 处占位符
2. **HIGH**：补充 7 个缺失章节
3. **HIGH**：修复 6 个概念占位符

**预期效果**：
- 修复后质量评分：预计 75-85/100
- API 成本：仅修复问题部分，约为全量重新生成的 20%

## 注意事项

1. **修复清单位置**：保存在 Obsidian vault 的 `.repair_manifests/` 目录
2. **修复标记**：添加到 Markdown 文件头部（Obsidian Callout 语法）
3. **API 成本**：自动修复会消耗 LLM API 额度，建议先人工审核清单
4. **多次修复**：部分问题（如章节合并）可能需要多次尝试
5. **人工审核**：CRITICAL 问题建议人工审核后再自动修复

## 故障排除

### Q: 修复清单未生成？

检查配置：
```yaml
quality_gate:
  lazy_write: true
  generate_repair_manifest: true
```

### Q: 修复后质量仍不达标？

1. 查看修复清单中的 `repair_status` 字段
2. 检查 `repaired_count` 是否为 0
3. 手动修复 CRITICAL 问题

### Q: 如何手动修复？

1. 打开修复清单 Markdown 文件
2. 根据字段路径定位问题字段
3. 手动编辑输出文件
4. 移除修复标记

## 总结

增量修复机制解决了批量处理时部分质量不合格导致全量重来的问题，通过：
- ✅ **先落盘**：质量不达标仍写入，避免浪费
- ✅ **修复清单**：自动生成优先级驱动的修复列表
- ✅ **增量修复**：只修复问题部分，降低成本
- ✅ **自动化**：`batch_repair.py` 支持批量自动修复
